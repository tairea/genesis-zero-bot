🌿⚡ **MESSAGE 2: THE CORE RESOLUTION PRINCIPLE**

One principle. Devastatingly obvious in hindsight:

---

**The system is not a machine to build. It's a garden to tend.**

---

Everything else follows from this.

Machines are built once, run until broken, then replaced. Gardens are living systems — you plant them, tend them, they grow in unexpected ways, some things die, new things emerge, and the gardener's job is to create conditions for life, not to control every outcome.

**What this changes:**

→不再追求"完美设计" — Instead of a perfect design that runs forever, you build a system that teaches its own repair.

→不再把故障当作失败 — Failure is not a bug to eliminate. Failure is information. The system learns from collapse, not despite it.

→不在层间做硬切换 — Instead of hard transitions between stages, you create overlapping redundancies so the next layer is always already partially present in the current one.

→不把人当资源 — People aren't resources to be allocated. They're the system. Their motivation, their relationships, their sense of meaning — these ARE the architecture.

→不分离社会与技术 — Social and technical systems aren't separate layers to integrate. They're the same thing at different scales of time.

**The practical implication:**

Design every component to:
1. Do its job (obviously)
2. Teach whoever maintains it how to build a replacement (teaching is load-bearing)
3. Fail in a way that teaches more than it destroys (graceful degradation)
4. Connect to at least 2 other components so failure cascades are visible, not silent

This is ECS applied to civilization. Entities are communities. Components are physical/social/knowledge. Systems are the processes that run across components. The formal model holds.

**The single question that guides every design decision:**

"If this breaks, what does the system learn?"

Not "how do we prevent breakage." That's impossible post-collapse. The question is: what does breakage teach?

Because a system that learns from failure survives. A system that hides failure doesn't.
